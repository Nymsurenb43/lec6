# Лекц №6 Сэдэв: Backup & Restore

## Backup гэж юу вэ?
Backup гэдэг нь таны чухал файл, өгөгдөл, системүүдийн хуулбарыг үүсгэж, өөр аюулгүй газар (гадаад диск, үүлэн технологи, сервер гэх мэт) хадгалахыг хэлнэ.

Зорилго: Техник хангамж эвдрэх, вирус халдах (Ransomware), эсвэл санамсаргүй байдлаар файлаа устгахаас сэргийлж "нөөц төлөвлөгөө" бэлдэх.

Төрлүүд:

Full Backup: Бүх өгөгдлийг бүхэлд нь хуулах.

Incremental Backup: Хамгийн сүүлийн нөөцлөлтөөс хойш өөрчлөгдсөн хэсгийг л хуулах.

Differential Backup: Хамгийн сүүлийн "Full" нөөцлөлтөөс хойш өөрчлөгдсөн бүх зүйлийг хуулах.


## 1 Backup ба Restore ойлголт

Backup (Нөөцлөлт): Энэ нь өгөгдөл алдагдах, гэмтэх үед ашиглах зорилгоор мэдээллийн хуулбарыг үүсгэж, тусдаа орчинд хадгалах үйл явц юм.

Restore (Сэргээлт): Энэ нь нөөцөлсөн хуулбараас өгөгдлийг буцааж системд оруулж, хэвийн үйл ажиллагаанд шилжүүлэх процесс юм.

## 3. MySQL Backup ба Restore
Өгөгдлийн сангийн хувьд backup-ыг дотор нь Logical болон Physical гэж ангилдаг.

Logical Backup (mysqldump): Өгөгдлийн санг SQL команд (CREATE TABLE, INSERT) хэлбэртэй текст файл болгон гаргахыг хэлнэ.

Давуу тал: Файлыг уншиж болдог, өөр хувилбарын MySQL рүү шилжүүлэхэд хялбар.

Сэргээх: mysql -u root -p db_name < backup.sql командаар сэргээнэ.

Physical Backup: Өгөгдлийн сангийн файлуудыг (.ibd, .frm г.м) шууд хуулах. Том хэмжээний өгөгдлийн санд илүү тохиромжтой.

## 4. Аюулгүй байдал ба Стратеги
Backup бол зөвхөн техникийн ажил биш, энэ бол Бизнесийн тасралтгүй ажиллагааны (BCP) салшгүй хэсэг юм.

Security (Аюулгүй байдал): * Encryption: Нөөцөлсөн файл дотор эмзэг мэдээлэл байдаг тул заавал нууцлах (encrypt) шаардлагатай.

Off-site storage: Нөөц хуулбарыг үндсэн серверээс өөр газар (Cloud эсвэл өөр дата төв) хадгалах.

Strategy (Стратеги):

3-2-1 дүрэм: 3 хувь хуулбар, 2 өөр төхөөрөмж дээр, 1 хувь нь заавал өөр байршилд байх.

RTO (Recovery Time Objective): Системийг хэр хурдан сэргээх шаардлагатай вэ?

RPO (Recovery Point Objective): Хамгийн ихдээ хэдэн минутын өгөгдөл алдахыг зөвшөөрөх вэ?




# ӨС-ийн нөөцлөлт болон сэргээхтэй холбоотой нийтлэг алдаанууд

1. Нөөцлөлтийг (Backup) шалгаж үзэхгүй байх
Хамгийн том бөгөөд хамгийн аюултай алдаа бол backup файл үүссэн л бол түүнийг ажиллаж байгаа гэж сохоор итгэх явдал юм.

Алдаа: Нөөцөлсөн файл нь гэмтэлтэй (corrupt) байх эсвэл дутуу бичигдсэн байх.

Шийдэл: Тогтмол хугацаанд (жишээ нь, сард нэг удаа) нөөц файлыг тусдаа орчинд Restore хийж, өгөгдөл бүрэн бүтэн байгааг шалгах туршилт хийх.

2. Нөөцлөлтийг үндсэн сервер дээрээ хадгалах
Олон хүн backup файлаа тухайн серверийнхээ өөр нэг хавтсанд (folder) хадгалаад орхидог.

Алдаа: Хэрэв серверийн хатуу диск (HDD/SSD) эвдрэх эсвэл сервер бүхэлдээ халдлагад өртвөл үндсэн өгөгдөл болон нөөц хуулбар хоёулаа хамт устана.

Шийдэл: 3-2-1 дүрмийг баримталж, дор хаяж нэг хуулбарыг үүлэн технологи (Cloud) эсвэл өөр газар байрлах серверт хадгалах.

3. Зөвхөн "Full Backup" ашиглах эсвэл буруу тохируулах
Өгөгдлийн сан томрох тусам зөвхөн бүтэн нөөцлөлт (Full Backup) хийх нь цаг хугацаа болон зай талбай их үрдэг.

Алдаа: Өдөр бүр хэдэн зуун GB файл үүсгэснээр дискний зай дүүрэх, нөөцлөлт хийх явцад системийн ачаалал эрс нэмэгдэх.

Шийдэл: Incremental эсвэл Differential backup-ыг хослуулан ашиглах стратегийг нэвтрүүлэх.

4. Transaction Log (Гүйлгээний түүх)-ийг үл тоомсорлох
MySQL-ийн Binary Log эсвэл MSSQL-ийн LDF файлуудыг нөөцлөхгүй байх нь эрсдэлтэй.

Алдаа: Хамгийн сүүлийн Full Backup-аас хойш гарсан өөрчлөлтүүдийг алдах.

Шийдэл: Point-in-Time Recovery (PITR) буюу тухайн цаг мөч рүү системээ буцааж сэргээх боломжийг бүрдүүлэхийн тулд лог файлуудыг тогтмол нөөцлөх.

5. Security буюу Нууцлал дутуу байх
Backup файл бол таны бүх өгөгдлийн "цээж хувилбар" юм. Хэрэв энэ файл алдагдвал хакерууд таны систем рүү нэвтрэх шаардлагагүйгээр бүх мэдээллийг тань авч чадна.

Алдаа: Нөөц файлыг нууцлаагүй (Encryption) хадгалах, эсвэл хандах эрхийг хэт нээлттэй орхих.

Шийдэл: Нөөцлөлтийг үүсгэх явцдаа шууд нууцлах (AES-256 гэх мэт) ба хандах эрхийг зөвхөн админ түвшинд хязгаарлах.

6. Хуучин нөөц файлуудыг устгахгүй байх (Retention Policy)
Алдаа: 2-3 жилийн өмнөх хэрэгцээгүй backup-уудыг хадгалсаар байгаад серверийн зайг дүүргэх. Энэ нь шинэ backup үүсэх боломжгүй болгодог.

Шийдэл: Автоматаар устгах бодлого (Retention Policy) тохируулах. Жишээ нь: Сүүлийн 7 хоногийн өдөр бүрийн backup, сүүлийн 1 сарын долоо хоног бүрийн backup-ыг хадгалах гэх мэт.
